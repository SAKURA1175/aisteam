# TutorMarket AI 当前进度与推进计划

## 1. 当前项目进度结论

截至当前代码状态，项目已经完成了 `一期 MVP 骨架`，不是从 0 开始。

可以明确认定已经完成的部分：

- `Next.js Web` 已搭起首页、老师市场页、登录页、聊天页
- `Spring Boot main-api` 已完成认证、老师、偏好、会话、知识库上传、审计等主业务骨架
- `FastAPI ai-orchestrator` 已完成健康检查、流式聊天、知识库 ingest/retry 接口
- 本地基础设施已具备 `PostgreSQL + Redis + MinIO + Docker Compose`
- 聊天主链路已打通到 `SSE` 流式返回
- Spring Boot 测试已通过
- Next.js 生产构建已通过

当前仍未完成或只完成了骨架的部分：

- 长期记忆前端页面
- 规则管理后台页
- 老师知识库管理页
- 真正的支付与订阅能力
- 小程序与移动端
- 完整监控、告警、运营看板

一句话判断：

当前代码状态已经具备“继续做产品”的基础，但还没有到“可试运营”的程度。

## 3. 当前主要问题

### 3.1 已发现问题

- `ai-orchestrator` 的 pytest 导入路径未配置，导致测试收集失败
- 当前 README 和真实代码进度之间需要持续同步
- 规则系统、知识库系统和聊天页虽然已有骨架，但后台管理闭环还不完整

### 3.2 架构风险

- `Spring Boot` 和 `FastAPI` 的职责边界后面必须文档化
- 长期记忆如果直接往现有会话链路里硬塞，后面会越来越乱
- 用户私有知识库如果不从数据层先设计隔离，会影响之后所有接口

### 3.3 产品风险

- 目前老师市场和聊天体验可展示，但“长期记忆”这个核心卖点还没有兑现
- 没有用户私有知识库之前，个性化程度还不够
- 没有支付和运营闭环，就还不能算真正试运营

## 4. 本轮继续推进建议

从当前状态往下推进，优先级应该是：

1. 修正测试和开发基线
2. 设计并接入长期记忆前端页与管理体验
4. 补后台页面
5. 再考虑商业化能力

不是先做：

- 小程序
- App
- 支付
- 开放老师入驻

## 5. 推荐下一阶段排期

## Phase A：开发基线稳定

周期：`2-3 天`

目标：

- 补齐 Python 测试执行环境
- 固化 `.env.example`
- 校准 README、启动文档和接口说明
- 跑通本地一键启动

验收标准：

- `mvn test` 通过
- `pytest` 通过
- `npm run build:web` 通过
- 新同学可按文档本地启动

## Phase B：用户私有知识库

周期：`4-6 天`

目标：

- 增加 user-scoped knowledge file 表结构
- 增加上传接口、列表接口、删除接口
- ai-orchestrator 检索时合并老师知识库和用户私有知识库
- 前端补“我的资料”页面

验收标准：

- 学生可上传私有文档
- 私有文档仅当前老师 + 当前用户可检索
- 回答引用能区分老师知识库和用户私有知识库

当前状态：`已完成后端主链路`

## Phase C：长期记忆 MVP

周期：`5-7 天`

目标：

- 新增 `memory_records` 表
- 定义记忆类型：画像、目标、偏好、进度、错误点
- 聊天结束后通过内部接口抽取稳定事实并回写主 API
- 新建会话时自动召回相关记忆
- 仍待前端增加记忆查看页

验收标准：

- 同一用户在同一老师下开启新会话，系统能记住目标和偏好
- 不同老师间记忆不互通
- 用户可查看并删除错误记忆

## Phase D：老师后台闭环

周期：`4-5 天`

目标：

- 老师规则版本页
- 知识库文件状态页
- ingest/retry 可视化
- 反馈查看页

验收标准：

- 运营可以不改代码调整老师规则
- 运营能看到知识库索引状态和失败原因

## 6. 建议分工

### 前端

- 1 人主做聊天页、记忆页、知识库页
- 1 人主做后台页和交互收口

### Java 后端

- 1 人主做用户私有知识库、权限、接口
- 1 人主做规则、后台接口、审计

### FastAPI / AI

- 1 人主做检索合并、记忆抽取、知识库 ingest 链路

如果只有 3 个人：

- 前端 1 人
- Java 1 人
- Python/AI 1 人

也能推进，但节奏要严格按优先级走。

## 7. 当前最推荐的下一步

如果只做一件最值的事，我建议现在就做：

`用户私有知识库 + 长期记忆 MVP`

原因：

- 这是和普通 AI 聊天产品拉开差距的核心能力
- 这两项补上后，产品才真正开始接近你最初的项目目标
- 后台、支付、小程序都可以后置
