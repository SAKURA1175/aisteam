# TutorMarket AI 前端计划书

## 1. 目标

前端系统负责承载以下核心能力：

- 老师市场展示
- 学生聊天与学习体验
- 老师配置后台
- 平台管理后台
- 多端分发，包括 Web、小程序和移动端

前端的目标不是只做一个聊天框，而是做出一个可运营、可扩展、可多端复用的产品层。

## 2. 前端总体策略

采用“多端分治，共享底层”的方案：

- Web 端：`Next.js`
- 小程序端：`Taro`
- iOS / Android：`React Native + Expo`
- 共享层：`TypeScript types + SDK + design tokens + 聊天协议`

不追求一套 UI 代码跑所有端，而是追求：

- 共享业务协议
- 共享接口定义
- 共享设计规范
- 各端保留合理差异

## 4. 技术选型

### 4.1 Web：Next.js

原因：

- 适合做内容页、后台、聊天页和 SEO 页面
- React 生态成熟
- 鉴权、路由、服务端渲染和中后台开发都很顺手

### 4.2 小程序：Taro

原因：

- 适合中国市场的小程序交付
- React 风格开发，和主前端栈一致
- 比直接把 Web 方案硬改成小程序更现实

### 4.3 App：Expo + React Native

原因：

- 小团队开发效率高
- 适合快速验证 iOS 和 Android
- 与 React 技术栈一致，复用认知成本低

## 9. 状态管理建议

- 服务端状态：`TanStack Query`
- 本地状态：`Zustand`
- 表单状态：`React Hook Form`

这样分工的原因：

- 接口缓存和重试交给 Query
- 页面级轻状态交给 Zustand
- 表单交互交给 React Hook Form

## 10. 国际化方案

- Web 使用 `next-intl`
- 小程序和 App 共享语言包
- 统一多语言 key 规则
- 所有老师、知识块、记忆项都带语言标记

## 13. 风险点

- 想用一套 UI 完全覆盖所有端，最后会非常痛苦
- 聊天页复杂度会比普通后台高很多
- 多语言和引用展示会增加组件复杂度
- 小程序与 Web 在文件上传、滚动、输入法上差异较大

## 14. 最现实的前端建议

- 第一期只把 Web 做好
- 第二期上小程序
- 第三期再做 App
- 不要一开始为了“全端”把前端节奏拖死
