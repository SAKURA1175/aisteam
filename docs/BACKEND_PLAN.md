# TutorMarket AI 后端计划书

## 1. 目标

后端负责搭建整个平台的核心业务能力：

- 用户与鉴权
- 老师系统
- 会话系统
- 长期记忆系统
- 知识库系统
- 规则系统
- 支付与订阅
- 后台与审计

这个项目的后端重点不是普通 CRUD，而是：

- 数据隔离
- AI 编排
- 检索链路
- 记忆写回
- 可运营和可审计

## 2. 后端总体架构

推荐采用“模块化单体 + 独立 AI 服务 + 异步任务系统”的方式。

### 2.1 主业务后端

推荐框架：`Spring Boot`

负责：

- 用户体系
- 鉴权与权限
- 老师配置
- 订单与订阅
- 会话记录管理
- 后台接口
- 审计日志

### 2.2 AI 编排服务

推荐框架：`FastAPI`

负责：

- 对话编排
- RAG 检索组合
- Prompt / Rules 组装
- 长期记忆抽取
- 评测任务

### 2.3 异步任务层

推荐组件：

- `Redis`
- `Celery` 或 `BullMQ`

负责：

- 文档解析
- 知识切片
- 索引构建
- 记忆抽取
- 重试与补偿

## 3. 为什么这么拆

- `Spring Boot` 适合企业级业务后端，权限、事务、支付、订单和管理系统更稳
- `FastAPI` 更适合 AI 工程、文档处理和 RAG 编排
- 异步任务必须独立，不然上传文档和记忆更新会拖慢主会话链路

这比“所有东西全塞一个后端项目里”更现实，也比“一开始就全微服务化”更稳。

## 5. 核心数据原则

### 5.1 隔离原则

核心隔离键统一为：

`tenant_id + teacher_id + user_id`

### 5.2 为什么必须这样

这是整个系统最关键的地方。  
如果老师之间、用户之间的记忆和知识库没有从数据层隔离，产品就不成立。

## 6. 推荐数据库设计

主数据库：`PostgreSQL`

原因：

- 强事务
- 结构化业务能力强
- 适合用户、老师、订单、日志、规则等主业务数据
- 支持 `pgvector`

建议核心表：

- users
- teachers
- teacher_rule_versions
- user_preferences
- conversations
- messages
- memory_records
- knowledge_files
- knowledge_chunks
- subscriptions
- payment_orders
- feedback_events
- audit_logs

## 7. AI 能力怎么接

### 7.1 对话能力

使用 OpenAI `Responses API`

### 7.2 检索能力

第一阶段使用 OpenAI `file_search`

### 7.3 长期记忆能力

采用三层设计：

- 当前会话摘要
- PostgreSQL 里的结构化长期记忆
- 语义检索辅助召回

### 7.4 关键实现原则

- 每轮请求后端都重新拼装规则
- 每轮请求都显式做记忆和知识检索
- 记忆写回走异步任务，不阻塞用户回复

## 8. 推荐请求链路

1. 用户发送消息
2. 主后端鉴权并校验订阅状态
3. 查询老师配置、用户偏好和规则版本
4. 调用 AI 编排服务
5. AI 编排服务检索长期记忆和知识库
6. 调用模型生成回答
7. 返回前端流式结果
8. 异步写会话摘要和长期记忆
9. 写审计日志和反馈指标

## 9. 异步任务链路

### Phase 2：AI 与记忆

### Phase 3：商业化

- 订阅支付

### Phase 4：扩展与稳定

- 监控
- 告警
- 性能优化
- 更多租户能力

## 13. 风险点

- 一开始就做微服务会过度设计
- 把记忆直接做成“聊天记录向量库”会失控
- 不做权限隔离，后期一定出事故
- 文件处理和索引失败补偿如果没设计，线上会很难维护

## 14. 最现实的后端建议

- 业务后端优先用 `Spring Boot`
- AI 服务单独用 `FastAPI`
- 数据先稳在 `PostgreSQL`
- 不要一开始引入过多中间件
- 先做模块化单体，跑稳后再拆
