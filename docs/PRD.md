# TutorMarket AI 产品需求文档

## 1. 项目概述

### 1.1 项目名称

TutorMarket AI

### 1.2 项目定位

TutorMarket AI 是一个面向教育场景的 AI 老师市场平台。每个 AI 老师都拥有：

- 独立的教学规则
- 独立的公开知识库
- 独立的学生记忆空间

每个学生还可以为某个老师附加自己的私有知识库和个性化偏好，使老师在开启新会话后仍能延续教学上下文。

### 1.3 核心价值

一个真正“记住学生、风格稳定、回答有依据”的 AI 教学市场，而不是普通聊天机器人换皮。

## 2. 背景与问题

当前大多数教育类 AI 产品存在以下问题：

- 新开会话后不记得用户
- 不同老师之间的人设不稳定
- 上传资料后权限隔离不清晰
- 回答更像泛问答，而不是围绕教学目标展开

教育市场里，`长期记忆` 和 `数据隔离` 不是加分项，而是底线。  
如果不同老师串了记忆，或者学生私有资料被其他老师或其他用户检索到，产品就会直接失去可信度。

## 3. 产品目标

### 3.1 商业目标

- 搭建一个可复用的 AI 教学平台，而不是单一聊天页面
- 支持未来多个 AI 老师、不同付费方式和市场化扩展
- 做出一个既能用于简历展示，也能用于真实试运营的产品

### 3.2 用户目标

- 学生和某位老师在新会话中继续学习，不需要重复介绍自己
- 不同老师拥有不同风格，且互不干扰
- 学生上传自己的资料后，回答能真正基于这些资料
- 学生可以调整回答偏好，但不能破坏平台安全边界

### 3.3 一期不做的内容

- 不做真人直播和排课
- 不做开放式教师自由入驻
- 不做全学科大而全平台
- 不做复杂社交和社区功能

## 4. 目标用户

### 4.1 核心用户

- 成人教育用户
- 语言学习用户
- 考试备考用户
- 编程学习用户

### 4.2 次级用户

- 内容运营人员
- 老师配置人员
- 平台管理员

### 4.3 最适合的起步细分方向

建议第一阶段只选一个垂直场景，例如：

- 雅思口语老师
- 考研英语老师
- 编程面试辅导老师

## 5. 核心产品原则

- 记忆必须可用、可编辑、可删除
- 知识库必须可隔离、可追溯、可调试
- 老师风格必须稳定，而不是每轮飘
- 回答必须尽量给出引用依据
- 跨端要做，但不能为了“全端”牺牲产品质量

## 6. 功能范围

### 6.1 老师市场

- 老师列表页
- 老师详情页
- 老师标签和分类
- 试听入口
- 订阅或购买入口

### 6.2 会话系统

- 新建会话
- 历史会话查看
- 多轮聊天
- 引用来源展示
- 消息点赞点踩
- 多语言输入输出

### 6.3 长期记忆系统

- 记录学生基础画像
- 记录学习目标
- 记录学习进度
- 记录常见错误和薄弱点
- 记录语言偏好与回答偏好
- 提供记忆查看、修改、删除能力

### 6.4 规则系统

- 平台级规则
- 老师级规则
- 用户级偏好设置
- 规则版本管理
- 规则回滚

### 6.5 知识库系统

- 老师公开知识库
- 学生私有知识库
- 文档上传与解析
- 文档切片与索引
- 元数据管理
- 检索权限控制

### 6.6 管理与运营后台

- 老师配置后台
- 知识库管理后台
- 规则管理后台
- 用户反馈处理
- 风险内容审核
- 系统运行指标看板

## 7. 核心用户故事

### 7.1 学生视角

- 作为学生，我希望老师记得我的目标、水平和偏好，即使重新开启会话也不丢失。
- 作为学生，我希望不同老师保持不同风格，并且彼此之间不共享我的信息。
- 作为学生，我希望上传我的笔记、教材、作文后，老师的回答能基于这些资料。
- 作为学生，我希望自己调整回答语言、详细程度和纠错力度。
- 作为学生，我希望看到引用来源，从而判断回答是否可信。

### 7.2 运营视角

- 作为运营人员，我希望可以配置老师规则，而不需要每次改代码。
- 作为运营人员，我希望上传知识库后能看到索引状态和检索效果。
- 作为运营人员，我希望通过数据比较不同老师的使用情况和留存表现。

### 7.3 管理员视角

- 作为管理员，我希望平台具备强隔离和审计日志，避免数据串用。
- 作为管理员，我希望能快速下线某个规则版本或知识源。
- 作为管理员，我希望查看风险输出和用户投诉记录。

## 8. 产品需求

### 8.1 记忆需求

- 同一学生在同一老师下开启新会话时，应能召回相关长期记忆
- 记忆隔离主键必须使用 `tenant_id + teacher_id + user_id`
- 长期记忆不能只依赖原始聊天记录
- 每条记忆应包含：置信度、更新时间、来源、删除标记
- 用户应能纠正错误记忆

### 8.2 规则需求

- 平台规则优先级最高
- 老师规则定义人格、教学方式、禁区和格式要求
- 用户仅能调整白名单项，例如回答语言、长短、是否严格纠错
- 规则必须可审计、可回滚、可灰度

### 8.3 知识库需求

- 老师知识库仅在该老师范围内可检索
- 学生私有知识库仅在“该学生 + 该老师”范围内可检索
- 检索支持按语言、来源、课程主题等元数据过滤
- 回答命中知识库时应尽量展示引用

### 8.4 多端需求

- 支持桌面端和移动端 Web
- 支持微信小程序
- 支持 iOS
- 支持 Android
- 核心聊天体验尽量保持一致

### 8.5 多语言需求

- 一期支持中文和英文
- 内容、记忆、知识块都带语言标签
- 回答语言可由用户偏好或当前会话上下文决定

## 9. 非功能需求

- 数据隔离必须可靠
- 对话延迟需控制在教育场景可接受范围内
- 索引流程需要支持失败重试和状态追踪
- 关键行为需要审计日志
- 架构需要支持未来多租户扩展

## 10. 推荐系统设计

### 10.1 核心服务

- API 网关
- 认证服务
- 对话编排服务
- 长期记忆服务
- 知识索引服务
- 检索服务
- 规则服务
- 支付服务
- 后台管理服务

### 10.2 推荐请求链路

1. 学生发送消息
2. 后端加载平台规则、老师规则、用户偏好
3. 加载最近会话摘要
4. 检索该老师下的长期记忆
5. 检索老师公开知识库
6. 检索学生在该老师下的私有知识库
7. 组装上下文并生成回答
8. 存储消息、反馈和链路追踪
9. 异步抽取稳定记忆并写回数据库

### 10.3 推荐记忆模型

- `profile_memory`：学生是谁
- `goal_memory`：学生要达到什么目标
- `progress_memory`：学生当前学到哪里
- `error_memory`：学生有哪些高频错误
- `preference_memory`：学生偏好什么语言、语气、长度

### 10.4 推荐隔离模型

默认隔离键必须是：

`tenant_id + teacher_id + user_id`

这个设计是整个产品最关键的底层原则，用来防止老师之间、用户之间的记忆和知识串用。

## 11. 技术方案建议

### 11.1 AI 层

- 使用 OpenAI `Responses API` 做对话编排
- 使用 OpenAI `file_search` 做 RAG MVP
- 使用 `PostgreSQL + pgvector` 做自建长期记忆服务

### 11.2 后端

- `NestJS` 作为主后端框架
- `PostgreSQL` 作为主数据库
- `Redis` 作为缓存与任务依赖
- `BullMQ` 负责异步任务
- `S3 / OSS` 兼容对象存储保存上传文件

### 11.3 前端

- `Next.js` 负责官网、学生 Web、老师后台、管理后台
- `Taro` 负责微信小程序
- `Expo + React Native` 负责 iOS 和 Android

### 11.4 工程化方式

- 全栈优先使用 TypeScript
- 使用 monorepo 统一管理
- 共享 SDK、类型定义、提示词模板和配置
- UI 不强求所有端完全复用

## 12. 为什么这么选

- `Next.js` 适合官网、后台、内容页和 Web 端聊天界面，生态成熟
- `Taro` 更适合微信小程序，不用硬把 Web 方案塞进小程序约束里
- `Expo` 是小团队做原生双端最快的现实方案
- `NestJS` 结构清晰，适合做模块化后端
- `PostgreSQL + pgvector` 最适合“结构化业务数据 + 语义召回”并存的场景
- OpenAI 托管检索能让你先验证产品，而不是先花大量时间造检索基础设施

## 13. 版本规划

### Phase 0：方向验证

周期：2 周

交付：

- 选定 1 个垂直场景
- 设计 3 个 AI 老师原型
- 做出初步 Prompt 和检索原型
- 准备评测数据集

### Phase 1：MVP

周期：4 周

交付：

- Web 聊天
- 老师市场页
- 老师公开知识库
- 登录系统
- 基础后台

### Phase 2：个性化能力

周期：4 周

交付：

- 长期记忆
- 学生私有知识库
- 老师规则中心
- 引用展示
- 记忆管理页

### Phase 3：分发与商业化

周期：4 周

交付：

- 微信小程序
- 支付能力
- 反馈与埋点分析
- 审计日志和风险处理

### Phase 4：扩展

周期：6 到 8 周

交付：

- iOS 和 Android
- 更多语言
- 老师入驻能力
- 分成和结算体系

## 14. 成功指标

- 次日留存
- 7 日留存
- 人均会话轮数
- 记忆命中认可率
- 引用覆盖率
- 满意度评分
- 付费转化率
- 索引成功率
- 数据串用事故必须为 0

## 15. 关键风险

- 错误记忆被长期保留
- 产品和普通聊天机器人差异不够大
- 数据隔离出错
- 文档量上来后检索质量下降
- 功能堆得太多，开发节奏失控
- 合规与风控处理滞后

## 16. 最现实的执行建议

- 不要一开始做全学科
- 不要一开始开放老师自由入驻
- 不要把原始聊天记录直接当长期记忆系统
- 不要为了省事追求“一套 UI 跑所有端”
- 先做一个垂直赛道和 3 个强老师原型

## 17. 适合学生团队的 MVP 定义

如果你的目标是做一个真正能写进简历、也能拿出去展示的项目，最合适的 MVP 是：

- 1 个垂直方向
- 3 个 AI 老师
- Web + 微信小程序
- 长期记忆
- 老师与用户知识隔离
- 可配置老师规则
- 引用展示
- 基础部署和数据埋点

这已经足够强，不需要一开始就把所有功能都做满。
